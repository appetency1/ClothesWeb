## 二、两大端 + 七大模块

### 前台用户端（5个模块）

| 模块 | 核心功能 | 关键规则 |
|------|---------|---------|
| **用户注册** | 填写用户名、密码、手机号、地址 | 用户名2-10字符；密码6-16字符；手机号11位规范格式；用户名和手机号不可重复 |
| **用户登录** | 用户名+密码 或 手机号+密码 | 登录成功生成Token返回前端；前台普通用户和管理员都能登 |
| **服装查询** | 首页展示、按类别/风格筛选、按名称模糊搜索 | 不指定条件则查全部 |
| **服装详情** | 查看名称、风格、类别、价格、可选尺码 | 尺码根据服装类别动态加载；可选尺码加入购物车 |
| **购物车** | 添加商品、查看列表、修改数量、删除商品、批量结算 | 同用户+同商品+同尺码自动累加数量，不重复添加 |
| **我的订单** | 提交订单、支付、确认收货、删除订单、按状态查看 | 状态严格流转：未支付→未发货→已发货→已收货；仅未支付/已收货可删 |
| **个人中心** | 查看个人信息、修改信息（含密码） | 修改时重新校验用户名和手机号唯一性 |

### 后台管理端（3个模块）

| 模块 | 核心功能 | 关键规则 |
|------|---------|---------|
| **服装管理** | 查看全部、多条件查询、上架、编辑、删除、图片上传 | 仅管理员可操作；图片上传用时间戳前缀防重名；支持.jpg/.png |
| **订单管理** | 查询全部订单、按用户姓名和状态筛选、发货 | 仅未发货订单可执行发货操作 |
| **用户管理** | 查看普通用户、按姓名/手机号查询、添加、编辑、删除 | 只管理role=2的普通用户，不碰管理员账号 |

## 三、数据库六张表

| 表名 | 作用 | 核心关联 |
|------|------|---------|
| **t_user** | 存储前台用户和后台管理员 | role字段区分身份（1管理员/2普通用户）|
| **t_clothes** | 存储所有服装信息 | type_id外键关联t_type |
| **t_type** | 存储服装类型（连衣裙、帽子、鞋子等）| 被t_clothes和t_size引用 |
| **t_size** | 存储各类型对应的尺码 | type_id外键关联t_type，实现"不同类别不同尺码" |
| **t_cart** | 存储用户购物车数据 | cloth_id关联t_clothes，user_id关联t_user |
| **t_order** | 存储用户订单数据 | user_id关联t_user；status字段控制状态流转 |

**设计要点：**
- 用户表用role字段实现"管理员/普通用户"合一存储
- 服装和尺码都通过type_id关联类型表，保证数据一致性
- 购物车表用cloth_id + cloth_size + user_id三字段联合判断重复
- 订单表的clothes_details用字符串存储商品快照，简化设计

## 四、系统三层架构

```
前端层（Vue）
    ↓ HTTP/JSON
控制层（Servlet）—— 接收请求、参数解析、调用业务层、返回JSON
    ↓
业务层（Service）—— 业务逻辑处理、事务控制、Token生成
    ↓
数据层（DAO）—— SQL执行、结果映射、数据库交互
    ↓
数据库（MySQL）
```

**配套机制：**
- **过滤器（AccessFilter）**：拦截需要登录的接口，校验请求头中的Token
- **工具类层**：数据库连接、Token生成、JSON转换等公共能力

## 五、核心设计亮点（答辩重点）

| 设计点 | 解决的问题 |
|--------|-----------|
| **Token无状态认证** | 登录后生成Token，前端每次请求携带，后端过滤器校验，不用Session，支持前后端分离 |
| **购物车合并逻辑** | 同一用户同一商品同一尺码自动累加数量，避免购物车重复条目 |
| **订单状态机** | 严格限制状态流转方向（0→1→2→3），防止非法操作（如直接跳到已收货）|
| **动态SQL拼接** | 服装查询、订单查询支持多条件组合（名称/类别/风格/状态），条件为空时不拼接，灵活且防注入 |
| **图片防覆盖** | 上传图片时加时间戳前缀，确保文件名唯一，避免不同商品图片互相覆盖 |
| **前后端分离** | 前端Vue独立运行，后端Servlet独立部署，通过标准化JSON接口通信，职责清晰 |

## 六、接口分类（按权限）

| 权限级别 | 接口范围 |
|---------|---------|
| **公开接口** | 注册、登录、查询服装、查看详情、模糊搜索 |
| **登录用户接口** | 购物车全部操作、订单全部操作、个人中心 |
| **管理员专用接口** | 服装管理、订单发货、用户管理 |

**权限控制方式：** 过滤器统一拦截，从请求头读取Token验证身份，管理员接口额外校验role=1。

## 九、关键预设问题

| 问题 | 回答核心 |
|------|---------|
| 为什么用Servlet不用SpringBoot？ | 教学项目，掌握底层原理，为后续学SpringBoot打基础 |
| Token怎么生成和验证？ | 用户ID+时间戳+随机数拼接后Base64编码；过滤器拦截解析 |
| 前后端怎么通信？ | Vue通过Axios/Fetch发HTTP请求，Servlet返回JSON |
| 怎么防止SQL注入？ | 全部用预编译语句（PreparedStatement），参数化查询 |
| 购物车为什么不做库存？ | 教学简化版，实际项目需加库存字段和乐观锁防超卖 |
